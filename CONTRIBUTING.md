# Contributing to OpenBlob

Thanks for your interest in contributing to OpenBlob.

OpenBlob is an open-source desktop copilot for Windows with a strong focus on local-first AI, useful automation, playful UX, and community-driven development.

## Ways to contribute

You do not need to contribute code only. Helpful contributions include:

- bug fixes
- refactors
- new commands and capabilities
- UI and UX improvements
- docs and architecture notes
- testing and debugging
- transcript and voice improvements
- connector integrations
- ideas, proposals, and feedback

## Before you start

For larger changes, please open an issue first so direction and scope can be aligned before implementation starts.

Small fixes, cleanup, documentation improvements, and focused bug fixes can usually be opened directly as a pull request.

## Development principles

When contributing, try to follow these principles:

- local-first where possible
- clear and understandable code over clever code
- small focused changes instead of huge mixed PRs
- preserve privacy-conscious behavior
- keep UX polished and intentional
- prefer extensible solutions over hardcoded one-offs

## Pull requests

Please try to keep pull requests:

- focused on one topic
- easy to review
- consistent with the current project structure
- tested manually if relevant

If your PR changes behavior, include a short explanation of:

- what changed
- why it changed
- anything reviewers should test

## Coding style

There is no heavy formal process yet, but please try to keep things:

- readable
- modular
- consistent with existing naming and structure
- easy for future contributors to understand

## Areas that are especially helpful right now

The project is still evolving, so especially useful contributions include:

- command routing improvements
- browser automation reliability
- transcript workflows
- voice pipeline improvements
- desktop UI polish
- multilingual support
- connector improvements
- contributor documentation
- bug fixing and cleanup

## Reporting bugs

When opening a bug report, include if possible:

- what you expected
- what actually happened
- steps to reproduce
- screenshots or logs if helpful
- environment details

## Feature ideas

Feature ideas are welcome. Please describe:

- the use case
- why it would help
- whether it affects UX, commands, memory, connectors, or architecture

## Questions

If something is unclear, open an issue or start a discussion first.
